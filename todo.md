## Removed from the backlog (with rationale)

Items previously parked here that we decided not to ship. Captured so we don't re-add them on autopilot.

- **Batch 5 тАФ MCP server.** Removed by user decision. Read-only MCP resource server would duplicate the existing native-skills install path for the 5 supported agents (Claude Code, Codex, Gemini, Cursor, Windsurf) and would either break the zero-runtime-deps invariant or require ~300 LoC of hand-rolled JSON-RPC. Defer until a real user reports needing BA Toolkit inside an agent that does not support native skills.
- **Batch 6 тАФ rename `/brief` тЖТ `/ba-toolkit.brief` (slash-command namespacing).** Premature optimisation against an unreported collision. Would break every existing user's muscle memory, every URL / doc / changelog reference, and require a 4.0.0 major bump plus a backward-compat alias on `/brief` for N versions. Re-open only if a real namespace collision with another skill pack is reported.
- **Batch 6 тАФ skill that validates a project against the generated artifacts (use-case compliance checker).** Research-grade idea: would need either per-language AST parsing (fragile, work per-stack) or LLM reasoning over a large codebase (slow, non-deterministic, no ground truth). The rest of BA Toolkit is about deterministic Markdown generation; this is the opposite direction. Re-open as a separate research spike if a feasible approach emerges.
- **Batch 6 тАФ skill that generates the tests required for TDD based on the produced artifacts.** Removed after the `/implement-plan` ship reframed the question. Three independent reasons: (1) **Duplicates existing artifacts** тАФ `05_ac_<slug>.md` is already test specifications in pure Given/When/Then form, and `/implement-plan` task `references` already point at the relevant AC. A separate `13_tdd_<slug>.md` would re-state the same information in a third place and require merge logic on every `/ac` or `/implement-plan` regenerate. (2) **Imposes a methodology** тАФ TDD is one valid testing strategy, but not every project does it (prototypes, spikes, internal tools, lightweight CLIs). A dedicated TDD skill biases the toolkit toward one process opinion. (3) **The right shape, if it ever becomes a real ask, lives in `/principles`, not in a new skill** тАФ extend `/principles` with a "Testing strategy: tdd / tests-after / manual / none" section and teach `/implement-plan` to conditionally enrich tasks with TDD-specific blocks based on that section. Same pattern as how `/principles` already governs language and ID conventions. **Do not build pre-emptively** тАФ wait for a real user request that says "my AI agent skips tests, I want the plan to enforce test-first". Until then, AC + DoD checkboxes in `/implement-plan` already give a TDD-discipline-friendly agent everything it needs.
