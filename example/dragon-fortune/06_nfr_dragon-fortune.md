# Non-functional Requirements: Dragon Fortune

**Slug:** dragon-fortune

## NFR-001: Spin API Response Time

**Category:** Performance
**Metric:** Spin API endpoint (`POST /api/v1/games/sessions/:id/spins`) must respond in ≤ 200 ms at the 95th percentile (p95) under normal load.
**Rationale:** Spin response directly impacts perceived gameplay smoothness. Values above 300 ms degrade the slot animation experience.
**Verification:** Load test with k6 at 500 concurrent users; p95 measured over a 10-minute sustained run.
**FR Reference:** FR-003
**Priority:** Must

---

## NFR-002: Concurrent User Capacity

**Category:** Scalability
**Metric:** The system must support 10,000 concurrent authenticated players without degradation of NFR-001 (p95 ≤ 200 ms for spins).
**Rationale:** Target MAU of 50,000 with a peak-to-MAU ratio of 20% implies ~10,000 concurrent users during peak hours.
**Verification:** Load test at 10,000 concurrent sessions; monitor p95 and error rate (must remain < 0.1%).
**FR Reference:** FR-002, FR-003
**Priority:** Must

---

## NFR-003: Platform Availability

**Category:** Availability
**Metric:** 99.9% uptime measured monthly (≤ 43.8 minutes of unplanned downtime per month). Planned maintenance windows excluded.
**Rationale:** iGaming players expect 24/7 access; extended downtime causes direct revenue loss and licence risk.
**Verification:** Uptime monitoring via external probe (e.g., Pingdom) with 1-minute check interval. Reported monthly.
**FR Reference:** FR-002, FR-003, FR-005, FR-006
**Priority:** Must

---

## NFR-004: Authentication Security

**Category:** Security
**Metric:** All API requests (except `POST /auth/telegram`) must be authenticated via JWT (HS256, 1-hour expiry). Telegram `initData` hash must be verified on every login attempt. Failed login attempts are rate-limited to 10 per IP per minute.
**Rationale:** Prevents unauthorised access, replay attacks, and credential stuffing.
**Verification:** Penetration test; automated token expiry test; rate-limit test.
**FR Reference:** FR-001
**Priority:** Must

---

## NFR-005: Data Encryption

**Category:** Security
**Metric:** All PII (player name, Telegram ID, KYC document references) must be encrypted at rest using AES-256. All client–server communication must use TLS 1.2 or higher.
**Rationale:** GDPR and iGaming licence requirements for personal data protection.
**Verification:** Infrastructure audit; TLS version scan (SSL Labs grade ≥ A); database encryption configuration review.
**FR Reference:** FR-001, FR-009
**Priority:** Must

---

## NFR-006: RTP Compliance

**Category:** Compliance
**Metric:** Actual RTP must remain within the certified range of 94.00%–97.00% calculated over a rolling window of 10,000 spins per game configuration. Deviations outside this range trigger an automatic alert to the Compliance Officer.
**Rationale:** Core iGaming licence requirement. Certified RTP is validated by the testing laboratory before launch.
**Verification:** Statistical sampling in staging (minimum 100,000 spins); automated compliance monitor in production with alerting.
**FR Reference:** FR-003, FR-015
**Priority:** Must

---

## NFR-007: AML and Responsible Gambling Compliance

**Category:** Compliance
**Metric:**
- KYC must be completed before withdrawals above 50 USD equivalent are processed (FR-009).
- Session time warnings must fire within 30 seconds of reaching the configured threshold (FR-012).
- Self-exclusion must take effect within 5 minutes of player confirmation (FR-013).
- Deposit limit enforcement must reject a deposit before calling the payment provider.
**Rationale:** Mandatory under the iGaming operating licence and applicable AML regulations.
**Verification:** Manual acceptance testing per checklist; automated regression test suite.
**FR Reference:** FR-009, FR-012, FR-013, FR-014
**Priority:** Must

---

## NFR-008: Test Coverage

**Category:** Maintainability
**Metric:** Unit + integration test coverage ≥ 80% on the following modules: spin calculation, RNG integration, wallet transaction processing, payment webhook handling, and bonus crediting logic.
**Rationale:** High-risk financial and gambling logic must have automated regression coverage to prevent silent regressions.
**Verification:** Coverage report generated by CI on every PR merge; coverage gate blocks merge if below threshold.
**FR Reference:** FR-003, FR-005, FR-006, FR-007, FR-011
**Priority:** Must

---

## NFR-009: Payment Processing Latency

**Category:** Performance
**Metric:** Deposit webhook processing (balance credit after provider confirmation) must complete within 5 seconds of receiving the webhook. Withdrawal status update must reflect within 60 seconds of provider response.
**Rationale:** Players expect near-instant balance updates after a successful deposit. Delays erode trust.
**Verification:** Integration test with payment provider sandbox; measure webhook-to-balance-update latency.
**FR Reference:** FR-005, FR-006, FR-007
**Priority:** Should

---

## NFR-010: Mini App Load Time

**Category:** Performance
**Metric:** The Mini App initial load (from Telegram launch to interactive slot screen) must complete in ≤ 3 seconds on a 4G connection (25 Mbps down, 10 Mbps up, 50 ms RTT).
**Rationale:** Telegram users have a low tolerance for slow Mini Apps; a 3-second load is the Telegram-recommended target.
**Verification:** Lighthouse performance audit in Telegram WebView emulation; manual test on physical iOS and Android devices.
**FR Reference:** FR-002
**Priority:** Should
